# 策略持续优化迭代算法系统 - Product Requirement Document

## Overview
- **Summary**: 设计并实现一套策略持续优化的迭代算法系统，该系统基于现有的选股策略和回测策略，通过"数据输入→选股→回测→策略优化→再选股"的完整迭代循环，自动优化策略参数和逻辑，形成可追溯的策略进化路径。
- **Purpose**: 解决策略优化依赖人工经验、效率低下、缺乏系统性迭代机制的问题，通过自动化的迭代优化系统持续提升策略表现。
- **Target Users**: 策略开发者、量化分析师、投资管理人员。

## Goals
- 实现历史数据起始点的灵活选择
- 集成选股策略生成推荐买入股票列表
- 集成回测策略进行收益计算与风险评估
- 实现基于回测结果的策略参数自动调整
- 形成完整的迭代循环流程
- 支持自定义迭代周期、策略调整幅度阈值及收敛条件
- 记录每次迭代的完整信息，形成可追溯的策略进化路径

## Non-Goals (Out of Scope)
- 不开发新的选股策略或回测策略，仅基于现有策略进行优化
- 不实现实时交易执行功能
- 不涉及策略部署和生产环境运行
- 不实现复杂的机器学习模型，优先使用基于规则和统计的优化方法

## Background & Context
- 现有系统已包含完整的选股策略框架（stock_selector模块）和回测引擎（src/core/backtest_engine.py）
- 回测服务（src/services/backtest_service.py）已支持完整的回测流程
- 策略管理器（stock_selector/manager.py）已支持策略倍率参数调整
- 需要将这些现有组件有机结合，形成自动化优化循环

## Functional Requirements
- **FR-1**: 历史数据起始点选择功能 - 支持用户指定或系统自动选择历史日期作为迭代起始点
- **FR-2**: 历史选股功能 - 基于指定历史日期的数据运行选股策略，生成推荐股票列表及买入价格
- **FR-3**: 历史回测功能 - 将选股结果输入回测引擎，计算收益指标和风险指标
- **FR-4**: 策略优化逻辑 - 根据回测结果自动调整策略参数（如策略倍率、板块阈值等）
- **FR-5**: 迭代循环控制 - 支持自定义迭代周期、调整幅度阈值、收敛条件
- **FR-6**: 迭代记录与追踪 - 记录每次迭代的策略参数、选股结果、回测指标、优化方向
- **FR-7**: 策略进化路径可视化 - 展示策略从初始状态到最终优化状态的完整进化过程

## Non-Functional Requirements
- **NFR-1**: 迭代效率 - 单次迭代（选股+回测+优化）应在合理时间内完成（<30分钟）
- **NFR-2**: 可扩展性 - 系统设计应支持未来添加更多优化算法和策略参数
- **NFR-3**: 可追溯性 - 所有迭代记录应完整保存，支持历史查询和复盘
- **NFR-4**: 容错性 - 单个迭代失败不应影响整个优化流程，应支持跳过或重试
- **NFR-5**: 配置灵活性 - 所有关键参数应支持通过配置文件或环境变量调整

## Constraints
- **Technical**: Python 3.10+，必须与现有代码风格和架构保持一致，使用 black+isort+flake8 代码规范
- **Business**: 必须基于现有的 stock_selector 模块和回测引擎进行开发，不能重写核心逻辑
- **Dependencies**: 依赖现有数据获取层（data_provider）、数据库存储（src/storage.py）、策略框架（stock_selector）、回测引擎（src/core/backtest_engine.py）

## Assumptions
- 假设历史数据已完整存储在数据库中，包括日线数据和分析历史
- 假设选股策略和回测引擎已稳定运行，无致命bug
- 假设用户有基本的Python和量化投资知识
- 假设迭代优化主要针对策略参数调整，暂不涉及策略逻辑重构

## Acceptance Criteria

### AC-1: 历史数据起始点选择
- **Given**: 用户启动策略优化系统
- **When**: 用户指定一个历史日期或让系统自动选择
- **Then**: 系统验证该日期有足够的历史数据，设置为迭代起始点
- **Verification**: `programmatic`
- **Notes**: 验证逻辑包括检查该日期及后续N天的日线数据是否存在

### AC-2: 历史选股执行
- **Given**: 已设置迭代起始点
- **When**: 系统运行选股策略
- **Then**: 生成该日期的推荐股票列表，包含每只股票的代码、名称、买入价格、策略匹配详情
- **Verification**: `programmatic`
- **Notes**: 选股结果应与该日期如果真实运行策略产生的结果一致

### AC-3: 历史回测执行
- **Given**: 已获取选股结果
- **When**: 系统运行回测引擎
- **Then**: 计算该股票组合在后续N天的收益指标（收益率、胜率、最大回撤等）和风险指标
- **Verification**: `programmatic`
- **Notes**: 回测窗口应可配置，默认使用现有系统的配置

### AC-4: 策略参数优化
- **Given**: 已获取回测结果
- **When**: 系统执行优化逻辑
- **Then**: 根据回测表现自动调整策略参数（如策略倍率、板块阈值等），调整方向应能改善回测指标
- **Verification**: `programmatic`
- **Notes**: 初始版本可先实现策略倍率的优化，基于回测收益率和胜率进行调整

### AC-5: 迭代循环控制
- **Given**: 已完成一次迭代
- **When**: 系统检查收敛条件
- **Then**: 如果未达到收敛条件且未超过最大迭代次数，继续下一次迭代；否则停止迭代并返回最优策略
- **Verification**: `programmatic`
- **Notes**: 收敛条件可包括：连续N次迭代收益变化<阈值、达到最大迭代次数等

### AC-6: 迭代记录保存
- **Given**: 每次迭代完成
- **When**: 系统执行记录逻辑
- **Then**: 保存该次迭代的所有信息：迭代编号、策略参数、选股结果、回测指标、优化方向
- **Verification**: `programmatic`
- **Notes**: 记录应持久化存储，支持后续查询

### AC-7: 进化路径展示
- **Given**: 已完成所有迭代
- **When**: 用户查看优化结果
- **Then**: 展示完整的策略进化路径，包括每次迭代的关键指标变化图表和参数调整记录
- **Verification**: `human-judgment`
- **Notes**: 可使用文本表格或简单的ASCII图表展示

## Open Questions
- [ ] 优化算法的具体实现：是使用简单的网格搜索、梯度下降，还是更复杂的贝叶斯优化？建议初始版本使用基于规则的启发式方法
- [ ] 迭代历史数据的存储位置：是使用现有数据库，还是单独的JSON/CSV文件？
- [ ] 多策略同时优化：是否支持同时优化多个策略？建议初始版本先支持单个策略优化
- [ ] 风险偏好设置：是否需要用户设置风险偏好（如保守型、激进型）来影响优化方向？
